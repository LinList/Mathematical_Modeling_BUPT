# 论文写作时间规划与避坑指南

## 本节导读

> 三天三夜的数学建模竞赛，时间是最稀缺的资源。论文成员如果不会规划时间，最后一天必然崩溃——摘要没时间写、排版没时间调、检查没时间做。
>
> 本节给你一份按小时安排的三天时间规划表，告诉你什么时间该干什么。再给你10个最常见的论文坑和避坑方法、一份20项的提交前检查清单、以及团队协作建议。照着做，你的论文就不会输在时间管理上。

---

## 一、三天比赛中的论文时间规划

### 总体原则

1. **从第一天就开始写**：不要等，能写什么先写什么
2. **摘要最后写但留足时间**：摘要需要所有结果，必须最后写，但至少留3-4小时
3. **先完成再完美**：先把框架搭起来、内容填进去，最后再精修
4. **每天和队友同步**：论文成员不能闭门造车，每天要跟进建模和编程进度

---

### Day 1：读题、列提纲、写问题重述和假设

| 时间 | 论文成员工作 | 备注 |
|------|------------|------|
| 08:00-10:00 | 和队友一起读题、讨论思路 | 三人必须一起，统一理解题意 |
| 10:00-11:00 | 列论文提纲 | 根据讨论结果，列出每个子问题对应什么模型、论文怎么组织 |
| 11:00-12:00 | 制作符号说明表初稿 | 和建模成员确认符号定义，第一天就统一符号 |
| 12:00-13:30 | 午休 | 休息很重要，三天是马拉松 |
| 13:30-15:00 | 写问题重述 | 用自己的话重新表述题目（详见01节） |
| 15:00-17:00 | 写问题分析 | 对每个子问题分析思路（详见01节） |
| 17:00-18:00 | 写模型假设 | 和建模成员确认假设条件，每条加理由 |
| 18:00-19:00 | 晚饭 | |
| 19:00-21:00 | 继续完善问题分析和假设 | 检查逻辑是否通顺 |
| 21:00-22:00 | 和队友开同步会 | 确认建模思路是否定了、编程是否开始了 |
| 22:00-23:00 | 设置论文排版框架 | 建好Word/LaTeX文件，设好格式，把已写内容放进去 |

**Day 1 产出**：论文提纲 + 问题重述 + 问题分析 + 模型假设 + 符号说明（初稿）

> **小贴士**：Day 1 最重要的是"搭骨架"。把论文框架搭好、能写的部分先写了，后面两天就只需要填充模型和结果。千万不要第一天无所事事地等队友。

| 时间 | 论文成员工作 | 备注 |
|------|------------|------|
| 08:00-09:00 | 和队友开晨会 | 确认建模进度，模型是否已经定型 |
| 09:00-12:00 | 写模型建立（问题一） | 让建模成员讲解模型思路，你用文字+公式写出来 |
| 12:00-13:30 | 午休 | |
| 13:30-14:00 | 跟进编程进度 | 确认编程成员是否有初步结果 |
| 14:00-17:00 | 写模型建立（问题二、三） | 继续写其他子问题的模型 |
| 17:00-18:00 | 整理编程成员的初步结果 | 如果有初步结果，开始做图表 |
| 18:00-19:00 | 晚饭 | |
| 19:00-21:00 | 完善模型建立部分 | 检查公式编号、符号一致性、文字解释 |
| 21:00-22:00 | 写模型求解部分（框架） | 先写算法步骤，结果位置留空 |
| 22:00-23:00 | 和队友开同步会 | 确认最终结果何时能出 |

**Day 2 产出**：模型建立（完整）+ 模型求解（框架）+ 初步图表

> **避坑指南**：Day 2 最怕的是"等"。如果建模成员的模型还没定型，你不要干等——先把已经确定的部分写了，不确定的先留框架。如果编程成员结果还没出，先把图表模板做好，结果一出就能填。

| 11:00-12:00 | 写模型求解结果分析 | 对每个结果做文字解读 |
| 12:00-13:00 | 午饭 | |

| 18:00-18:30 | 晚饭（快速吃） | |

> **避坑指南**：Day 3 是最紧张的一天。最常见的问题是"摘要来不及写"——因为结果出得晚。所以一定要在Day 3上午就催编程成员出最终结果。如果12点前结果还没出，先用估算值写摘要框架，结果出来后替换数字。

## 二、10个最常见的论文坑及避坑方法

### 坑1：摘要没有结果数字

**表现**：摘要里只有"得到了评价结果""预测精度较高"，没有任何具体数字。

**后果**：评委无法判断你做出东西没有，直接低分。

**避坑方法**：摘要里每个子问题至少给1-2个关键数字。写完摘要后检查：有没有数字？没有就回去补。

---

### 坑2：公式堆砌，没有文字解释

**表现**：模型建立部分连续放了五六个公式，中间没有文字，评委不知道每个公式在干嘛。

**后果**：评委看不懂你的建模过程，认为你在凑公式。

**避坑方法**：每个公式前后都要有文字。公式前说"为什么用这个公式"，公式后说"这个公式算的是什么、符号代表什么"。

---

### 坑3：图表没有编号或没有引用

**表现**：论文里有图有表，但没有编号（图1、表1），或者正文中没有引用。

**后果**：评委不知道你为什么放这个图表，也找不到对应位置。

**避坑方法**：每个图表必须编号，正文中必须引用（"如图1所示"）。最后检查时逐个核对。

---

### 坑4：符号前后不一致

**表现**：符号说明里用 $w_j$，正文里写 $W_j$；或者同一个符号在不同地方含义不同。

**后果**：评委觉得你不严谨，且容易造成理解混乱。

**避坑方法**：第一天就定好符号表，全篇严格遵守。最后检查时把符号表和正文逐项对照。

---

### 坑5：模型建立和问题分析重复

**表现**：问题分析里写了详细的方法步骤，模型建立里又写一遍，内容高度重复。

**后果**：论文冗长，评委觉得你在凑字数。

**避坑方法**：问题分析讲"为什么这么做、大致怎么做"（不写公式），模型建立讲"具体怎么做"（有公式有步骤）。两者分工明确。

---

### 坑6：结果不合理但不解释

**表现**：模型算出来某月空气质量得分为负数，或者预测AQI为500，你不解释为什么。

**后果**：评委认为你的模型有错误，直接扣分。

**避坑方法**：如果结果异常，要么修正模型，要么在论文中解释原因（如"由于该月发生沙尘暴，污染物浓度异常升高"）。不要假装没看见。

---

### 坑7：最后一天才开始写论文

**表现**：前两天光看队友建模编程，自己啥也没写，第三天开始疯狂赶。

**后果**：论文质量极差，摘要来不及写，排版没时间调。

**避坑方法**：从第一天就开始写！能写什么写什么——问题重述、问题分析、假设、符号说明第一天就能写。模型建立等模型定型了就写，不要等。

---

### 坑8：页数超标

**表现**：国赛要求不超过25页，你写了35页。

**后果**：超页部分评委可能不看，且认为你不会精简。

**避坑方法**：控制每个部分的篇幅。模型建立详细写，但基础知识不要展开讲（评委都懂）。代码放附录不计页数。如果超了，删掉不必要的内容。

---

### 坑9：参考文献格式混乱

**表现**：参考文献格式不统一，有的有期刊名有的没有，标点符号五花八门。

**后果**：评委觉得你不规范，印象分降低。

**避坑方法**：统一用GB/T 7714格式。写完后逐条检查格式是否一致。

---

### 坑10：没有模型检验

**表现**：模型算出结果就结束了，没有敏感性分析、误差分析、合理性验证。

**后果**：评委无法判断模型是否可靠，认为你的工作不完整。

**避坑方法**：模型检验与分析是论文的必要部分。至少做一个合理性验证（结果是否符合实际），预测类模型必须做误差分析，评价类模型最好做敏感性分析。

## 三、论文检查清单（提交前必查的20项）

提交前，逐项打勾。**任何一项没做到，都要修改后再提交。**

### 内容检查（10项）

| 序号 | 检查项 | ☐ |
|------|--------|---|
| 1 | 摘要包含五要素：背景+思路+方法+结果+结论 | ☐ |
| 2 | 摘要中每个子问题都有具体结果数字 | ☐ |
| 3 | 三个（或多个）子问题都完整解决，没有遗漏 | ☐ |
| 4 | 每个公式都有文字解释，没有公式堆砌 | ☐ |
| 5 | 有模型检验与分析（敏感性/误差/合理性） | ☐ |
| 6 | 模型评价有优缺点和改进方向 | ☐ |
| 7 | 摘要中的方法和结果与正文一致 | ☐ |
| 8 | 结果合理，异常结果有解释 | ☐ |
| 9 | 参考文献格式统一（GB/T 7714） | ☐ |
| 10 | 没有明显错别字和语病 | ☐ |

### 格式检查（6项）

| 序号 | 检查项 | ☐ |
|------|--------|---|
| 11 | 页边距、行距、字体字号符合规范 | ☐ |
| 12 | 各级标题格式统一 | ☐ |
| 13 | 公式连续编号，正文引用与编号一致 | ☐ |
| 14 | 图表连续编号，正文引用与编号一致 | ☐ |
| 15 | 图题在图下、表题在表上 | ☐ |
| 16 | 表格使用三线表格式 | ☐ |

### 排版检查（4项）

| 序号 | 检查项 | ☐ |
|------|--------|---|
| 17 | 目录已更新，页码与正文一致 | ☐ |
| 18 | 图片清晰（≥300dpi），不模糊 | ☐ |
| 19 | 符号说明与正文符号一致 | ☐ |
| 20 | 总页数不超过竞赛要求（国赛≤25页） | ☐ |

> **小贴士**：把这张清单打印出来，最后两小时逐项检查。不要相信"我觉得没问题"——一定要逐个看。很多低级错误（编号对不上、目录没更新）都是因为最后没检查。

## 四、团队协作建议

### 论文成员如何高效获取建模和编程的结果

论文成员的输入来自队友，如果获取信息效率低，论文就写不快。以下是高效协作的建议：

#### 1. 给队友一个"信息需求清单"

不要每次去问"你那个模型好了没"，而是给队友一个明确的清单，让他们按格式给你信息。

**给建模成员的清单：**
```
我需要以下信息来写论文，请按这个格式给我：
□ 模型核心思路（一段话，不超过100字）
□ 关键公式（写出公式+每个符号的含义）
□ 模型假设条件（列出每条假设+理由）
□ 参数设置（用了什么参数，值是多少，为什么）
□ 模型的适用条件和局限性
```

**给编程成员的清单：**
```
我需要以下信息来写论文，请按这个格式给我：
□ 运行结果数据（用Excel给我，列名要清楚）
□ 关键图表（300dpi以上，有标题）
□ 算法步骤（用1-2-3列出，不超过10步）
□ 核心代码（放附录用，加注释）
□ 运行环境（Python版本、用了什么库）
```

> **小贴士**：把这个清单第一天就发给队友，让他们知道你需要什么。这样他们工作时就有了"输出格式"的意识，给你的东西会更规范。

#### 2. 每天开两次同步会

- **早会（8:00-8:15）**：每人说昨天做了什么、今天打算做什么、有没有遇到问题
- **晚会（22:00-22:15）**：确认今天进度、明天计划

同步会不要长，15分钟够了。关键是保持信息畅通。

#### 3. 共享文档协作

- **Word**：用OneDrive或腾讯文档共享，但注意不要同时编辑（容易冲突）
- **LaTeX**：用Overleaf，支持多人实时协作（强烈推荐）
- **数据/图表**：用共享文件夹（如QQ群文件、OneDrive），统一存放

#### 4. 论文成员的主动沟通技巧

| 场景 | 错误做法 | 正确做法 |
|------|---------|---------|
| 想知道建模进度 | "你模型好了没？" | "你现在做到哪一步了？模型思路定了吗？能给我讲讲吗？" |
| 发现结果不合理 | "你这结果不对吧" | "我看了一下结果，这个数值好像偏大，你帮我看看是不是正常？" |
| 编程结果出得慢 | "你怎么还没出结果" | "你现在卡在哪一步？需要我帮忙吗？最晚什么时候能出？" |
| 需要队友解释公式 | "这个公式什么意思" | "我写论文需要解释这个公式，你能用一段话帮我说明一下吗？" |

> **避坑指南**：论文成员最容易犯的沟通错误是"被动等待"。不要等队友给你送信息，要主动去要。你是论文的"总编辑"，你有权知道所有进展。

#### 5. 应对突发情况的预案

| 突发情况 | 应对方案 |
|---------|---------|
| 编程成员结果出不来 | 先用估算值/模拟数据写论文框架，结果出来后替换 |
| 建模成员临时换模型 | 评估换模型的时间成本，如果时间不够，坚持原方案 |
| 模型结果不合理 | 和建模成员一起排查，如果是模型问题就修，如果是数据问题就解释 |
| 论文页数超标 | 删减不必要的内容（如冗长的公式推导、基础知识介绍） |
| 最后发现摘要和正文不一致 | 以正文为准修改摘要，确保完全对应 |

## 本节总结

### 时间管理三原则

1. **第一天就开始写**——不要等，能写什么先写什么
2. **摘要最后写但留3-4小时**——摘要需要所有结果，但值得反复打磨
3. **最后2小时检查**——用20项检查清单逐项核对

### 避坑三原则

1. **摘要必须有数字**——没有数字的摘要等于没写
2. **公式不能裸奔**——前后必须有文字
3. **结果必须检验**——没有检验的模型不可信

### 协作三原则

1. **给队友明确的清单**——不要模糊地问"好了没"
2. **每天两次同步会**——保持信息畅通
3. **主动沟通**——论文成员是总编辑，要主动要信息

---

> **最后一句心里话**：三天三夜的比赛很苦，但也是大学里最有成就感的经历之一。作为论文成员，你是团队的"最后一道关"——所有的努力最终都通过你的论文呈现给评委。把时间规划好、把坑避开、把检查做到位，你就对得起自己和队友这三天的付出。加油！
